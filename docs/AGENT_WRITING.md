# AI 写作规范

本文档用于约束 AI 协作者编写和修改博客文章的方式。当前项目已经迁移到 Next.js，文章仍放在 `posts/` 中，图片写法见 [
`IMAGE_USAGE.md`](./IMAGE_USAGE.md)。

## 1. 总体定位

技术文章应写成整理后的技术笔记，不写成教材复述、营销说明或泛泛散文。

目标：

- 主干清楚
- 信息密度高
- 边界明确
- 对后续代码、系统理解或问题排查有直接帮助

判断标准是“对象、关系、机制、接口是否讲清楚”，不是“是否把教材内容全部搬进来”。

## 2. 写作范围

修改文章前先确认文章类型：

| 类型             | 写法                          |
|----------------|-----------------------------|
| 技术笔记           | 聚焦对象关系、机制、接口、边界             |
| API / POSIX 文章 | 按接口对象、生命周期、参数、返回值、示例组织      |
| 迁移 / 更新记录      | 记录问题、处理方案、验证结果和遗留项，允许少量个人语气 |
| 个人记录 / 随笔      | 保留原作者语气，不套用过硬的技术笔记结构        |

不要把一种文章类型的写法强行套到另一种文章上。

## 3. 结构规则

文章结构应服务理解顺序：

1. 本篇讨论什么
2. 关键对象或问题
3. 核心机制
4. API、实现或操作步骤
5. 边界、常见问题或验证
6. 小结

要求：

- 正文标题、frontmatter `outline` 和锚点保持一致。
- 标题锚点使用 Markdown 标准写法 `{#slug}`，追加在标题末尾，例如 `## 基础数据类型{#基础数据类型}`；不要新增 `<a id=...></a>`。
- 每篇开头说明范围，不在小结里引入新主题。
- 子篇不要重复总览文章已经讲清的背景。
- 如果提纲列出若干子点，正文不能只用一句话带过。

## 4. 语言规则

### 4.1 禁用设问式标题

标题直接命名对象或机制。

| 禁用          | 使用         |
|-------------|------------|
| 什么是 POSIX？  | POSIX 的定义  |
| 为什么需要 VFS？  | VFS 的设计动机  |
| fork 复制了什么？ | fork 的资源复制 |

设问句也不要放在文章开头和表格左列。

### 4.2 少用说明文支架

下列表达优先删除或改成直述：

| 避免              | 替代          |
|-----------------|-------------|
| 可以理解成 / 可以看作    | 直接写“X 是 Y”  |
| 如果从……角度看        | 删除支架，直接进入判断 |
| 需要注意的是 / 值得注意的是 | 直接写结论       |
| 后面会看到 / 后续再展开   | 直接引用具体文章或删除 |
| 可以简要概括如下        | 直接给表格或结论    |

“不是……而是……”只在澄清常见误解时使用，不要作为常规起手式。

### 4.3 减少 AI 味

- 不写空泛总起句，例如“在现代软件开发中，X 非常重要”。
- 不写没有信息量的过渡句，例如“接下来我们将深入探讨”。
- 不用连续排比包装简单事实。
- 不把每节写成“定义、特点、优缺点、总结”的模板。
- 保留必要的人类写作节奏：短句、明确判断、具体例子。

## 5. 信息组织

### 5.1 写对象关系

每节落到具体对象，不堆概念名词。

示例：

- 用“`shm_open` 创建内核对象，`mmap` 建立进程虚拟地址映射”。
- 不用“共享内存是一种快速 IPC 方式”作为唯一解释。

### 5.2 表格用于稳定比较

适合用表格的内容：

- 类型比较
- 参数与返回值
- 生命周期
- 对象层级
- 适用场景
- 验证矩阵

表格必须有比较维度，不要只堆名词。

### 5.3 理论段落短而硬

技术段落优先采用：

1. 首句给判断
2. 中间用表格、代码或公式展开
3. 尾句说明边界或过渡

已经说清的判断不要换一种说法重复。

## 6. API 与代码示例

API 文章应把“接口 - 对象 - 生命周期 - 示例”串起来。

每个核心 API 至少包含：

- 一句作用摘要
- 原型
- 参数表
- 成功 / 失败返回值
- 使用边界

示例代码要求：

- 示例前用 1-2 句说明演示目标。
- 优先单文件闭环，可直接运行。
- 超过 30 行的示例在关键操作处补注释。
- 注释解释机制，不解释显而易见的语法。
- 短示例中的 `print` 输出优先写在对应代码行注释里，不单独增加“输出”代码块；只有复杂多行输出才单独列出。
- 如果用 `sleep` 稳定输出，必须说明它不是同步手段。

## 7. 图片、公式与代码块

- 图片引用遵循 [`docs/IMAGE_USAGE.md`](./IMAGE_USAGE.md)。
- 新文章图片使用 ```` ```ts image-setup ```` 与 `<Image {...imageConfig} />`。
- 公式只用于压缩关键关系，不用公式堆砌正文。
- 代码块保留语言标识，便于 Shiki 高亮。
- Markdown 标题锚点使用 `{#slug}`，并和 frontmatter `outline` 同步维护。

## 8. 技术系列补充规则

技术系列文章优先保留：

- 支撑后续接口的理论
- 对对象关系有澄清作用的层级划分
- 容易误解但工程上必须明确的边界

典型边界：

- 共享不等于同步
- 信号是通知，不是数据通道
- 管道是字节流，不保留消息边界
- `shm_unlink` 删除名字，不是立即回收对象
- `kill` 不只表示杀死进程

## 9. 更新记录写法

博客更新记录不必写成纯技术规范文档。推荐结构：

1. 原始问题
2. 处理方案
3. 验证结果
4. 遇到的问题
5. 收尾状态

可以保留少量个人语气，但不要影响事实清晰度。部署、测试、兼容性结论要写具体。

## 10. 修改前检查

修改文章前先确认：

- 是否需要同步 `outline`。
- 是否涉及图片引用规范。
- 是否会影响已有链接 slug。
- 是否需要运行 `pnpm check:post-images`、`pnpm typecheck` 或 `pnpm build`。

不要为“统一风格”批量改动无关文章。
