一篇围绕 Unix 进程部分展开的总览笔记。

---

在 [概述](./operating-sys-0-overview.md) 里，已经把操作系统拆成了进程、内存、文件、I/O 这些大块内容。这里开始进入“进程”这一章。

这一篇不展开 `fork`、`exec`、调度算法、线程库实现、同步原语或进程间通信，而是先建立整章会反复出现的前置信息：进程是什么、它拥有哪些资源、操作系统如何用
`PCB` 管理它、`pid / ppid` 和进程状态分别表示什么。

## 这一章关注什么<a id=这一章关注什么></a>

### 1. 进程视角下的几个核心问题<a id=进程视角下的几个核心问题></a>

把“进程”作为一章单独拿出来，通常是为了回答下面这些问题：

| 问题                | 关键对象                    |
|-------------------|-------------------------|
| 一个程序如何变成可运行实体     | 进程、地址空间、执行上下文           |
| 操作系统如何识别和管理它      | `pid`、`ppid`、`PCB`、进程状态 |
| 进程如何被创建、替换和回收     | `fork`、`exec`、`wait`    |
| 一个进程内部为什么还能有多条执行流 | 线程                      |
| 多个执行实体如何共享 CPU    | 调度                      |
| 多个执行实体如何协作与交换数据   | 同步、IPC                  |

本篇先处理前两类问题：建立进程抽象，并说明操作系统如何记录和管理进程。后面几篇文章再分别展开 API、线程、调度、同步和通信。

## 进程是什么<a id=进程是什么></a>

### 1. 程序与进程<a id=程序与进程></a>

程序是静态代码，进程是正在运行的程序实例。

磁盘上的可执行文件只是指令和数据的组织形式；只有当它被装入内存、分配运行所需资源并开始执行时，它才成为进程。同一个程序文件可以同时对应多个进程，而这些进程彼此独立，各自维护自己的运行状态。

可以压缩成下面这张表：

| 对象 | 含义        |
|----|-----------|
| 程序 | 静态代码与数据   |
| 进程 | 正在运行的程序实例 |

### 2. 进程在操作系统中的角色<a id=进程在操作系统中的角色></a>

在这一章里，可以先把进程看成两个层面的结合：

| 层面    | 作用                |
|-------|-------------------|
| 资源环境  | 表示“这个运行实体拥有什么”    |
| 执行上下文 | 表示“这个运行实体当前执行到哪里” |

从这个角度看，进程首先是资源容器，也是操作系统管理程序运行的基本对象。教材里常说进程是资源分配和调度的重要单位，强调的就是这层含义。

### 3. 进程与线程的关系<a id=进程与线程的关系></a>

后面会看到，线程出现后，真正被内核直接调度的往往是线程而不是整个进程。但在线程还没展开之前，先保留下面这个区分就够了：

| 对象 | 更接近什么      |
|----|------------|
| 进程 | 资源容器与管理单位  |
| 线程 | 进程内部的一条执行流 |

也就是说，进程回答的是“资源归谁”，线程回答的是“谁在执行”。本篇先把进程这个容器讲清楚，线程部分留到后文单独展开。

## 进程的基本组成<a id=进程的基本组成></a>

### 1. 资源环境<a id=资源环境></a>

从资源视角看，一个进程通常包含下面这些内容：

| 组成部分   | 作用                          |
|--------|-----------------------------|
| 地址空间   | 规定这个进程可访问的虚拟内存范围            |
| 代码段    | 保存要执行的机器指令                  |
| 数据段    | 保存全局变量、静态变量等运行时数据           |
| 堆      | 保存动态分配得到的内存                 |
| 打开的文件  | 保存当前进程已打开的文件、管道、终端、socket 等 |
| 信号处理方式 | 保存进程级别的信号处理配置               |

这些内容共同构成了进程的资源环境，也就是“这个进程拥有什么”。

### 2. 执行现场<a id=执行现场></a>

从执行视角看，一个进程还需要一组能恢复运行的现场信息：

| 组成部分    | 作用             |
|---------|----------------|
| 程序计数器   | 指出下一条要执行的指令    |
| CPU 寄存器 | 保存当前计算状态       |
| 栈       | 保存函数调用过程中的局部状态 |
| 调度状态    | 表示当前是运行、就绪还是等待 |

这部分内容更接近“执行现场”。后面讨论上下文切换时，本质上就是保存和恢复这组状态。

把前两个小节合起来，可以得到一个更稳定的对照：

| 层次   | 典型内容                 | 关注点         |
|------|----------------------|-------------|
| 资源环境 | 地址空间、代码段、数据段、堆、打开的文件 | 这个进程拥有什么    |
| 执行现场 | 程序计数器、寄存器、栈、调度状态     | 这个进程当前执行到哪里 |

### 3. 文件描述符表<a id=文件描述符表></a>

Unix 里有一个特别重要的进程资源，就是文件描述符表。

它记录当前进程打开了哪些对象，而这些对象不只包括普通文件，也包括终端、管道、socket 和设备文件。后面学习 `open`、`read`、`write`
、重定向、管道和进程间通信时，都会不断回到这张表。

程序刚启动时，通常已经默认打开了 3 个文件描述符：

| 名称       | fd  | 默认含义 |
|----------|-----|------|
| `stdin`  | `0` | 标准输入 |
| `stdout` | `1` | 标准输出 |
| `stderr` | `2` | 标准错误 |

## PCB：操作系统如何管理进程<a id=pcb操作系统如何管理进程></a>

如果系统里同时运行很多进程，操作系统就必须有一个地方记录“这个进程是谁、当前处于什么状态、执行到哪了、拥有哪些资源”。这个记录结构通常就叫
`PCB`，也就是进程控制块（`Process Control Block`）。

可以把 `PCB` 理解成“操作系统为每个进程建立的一份管理档案”。有了它，内核才能识别、调度、挂起、恢复和回收一个进程。

### 1. pid 与父子进程关系<a id=pid与父子进程关系></a>

每个进程都有一个进程 ID，也就是 `pid`。它是操作系统为该进程分配的标识符。

进程通常还会记录父进程 ID，也就是 `ppid`，表示“是谁创建了我”。后面学习 `fork` 时会看到，父进程调用 `fork` 之后，会产生一个新的子进程。

父子关系会直接影响生命周期中的几个特殊状态：

| 情况   | 含义                       |
|------|--------------------------|
| 僵尸进程 | 子进程已经结束，但父进程还没有 `wait` 它 |
| 孤儿进程 | 父进程先结束，子进程仍然继续运行         |

`wait` 的意义之一，就是把已经结束的子进程从“僵尸状态”推进到真正被系统回收的状态。

### 2. 进程状态与生命周期<a id=进程状态与生命周期></a>

进程并不是一创建出来就一直运行到结束。典型生命周期会在几种状态之间切换：

| 状态      | 含义                 |
|---------|--------------------|
| 新建      | 进程刚被创建，还未真正进入运行阶段  |
| 就绪      | 已经具备运行条件，只是在等待 CPU |
| 运行      | 当前正在 CPU 上执行       |
| 等待 / 阻塞 | 正在等待某个事件，例如 I/O 完成 |
| 终止      | 进程执行结束，等待资源回收      |

一个典型流转过程大致如下：

```text
新建 -> 就绪 -> 运行 -> 终止
              |
              v
             等待
              |
              v
            就绪
```

这些状态后面会和调度里的就绪队列、等待队列直接对应起来。

### 3. PCB 的典型组分<a id=pcb的典型组分></a>

一个 `PCB` 中通常会记录下面这些信息：

| 类别      | 典型内容                  |
|---------|-----------------------|
| 标识信息    | `pid`、`ppid`、用户 ID    |
| 进程状态    | 新建、就绪、运行、等待、终止等       |
| 程序计数器   | 下一条要执行的指令位置           |
| CPU 上下文 | 通用寄存器、栈指针等            |
| 调度信息    | 优先级、时间片、队列指针          |
| 内存信息    | 地址空间、页表、代码段 / 数据段相关信息 |
| I/O 信息  | 打开的文件、设备、文件描述符表       |
| 统计信息    | 已使用 CPU 时间、创建时间等      |

这里最重要的不是把字段逐项背下来，而是抓住 `PCB` 的作用：

- 让内核知道“这是谁”
- 让内核知道“它现在处于什么状态”
- 让内核在切换时保存和恢复执行现场
- 让内核在结束时回收与它相关的资源

因此，上下文切换从概念上看，就是把当前进程的执行现场保存到它的 `PCB`，再从下一个进程的 `PCB` 恢复对应现场。
