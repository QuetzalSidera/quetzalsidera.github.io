这份文档用于总结操作系统系列文章的行文规范，后续写作时统一参考。

## 总体定位

操作系统文章要写成“整理后的技术笔记”，而不是教材复述或泛泛说明文。目标是：

- 主干清楚
- 信息密度高
- 边界明确
- 能直接服务后续代码与系统理解

判断标准不是“是否把教材都讲到了”，而是“是否把这一主题最关键的对象、关系、机制和接口讲清楚了”。

## 总体原则

### 1. 先写主干，再决定是否保留枝节

正文应优先保留主题的主干机制，再决定是否保留引例、历史背景、经典算法或教材细节。

例如：

- 同步篇应优先保留竞争条件、执行顺序、锁/信号量/条件变量、饥饿、优先级反转、死锁
- 不必为了“完整”强行保留 Peterson、临界区构成这类对后续代码帮助较小的内容

如果某一部分不能直接提升理解主线、不能支撑后续 API、或只是教材推导过程，应优先删去。

### 2. 写“对象关系”，不要写“概念堆砌”

每一节都应落到明确对象上，而不是停留在抽象口号。

例如：

- 不只写“共享内存很快”，而要写“`shm_open` 创建内核对象，`mmap` 建立进程虚拟地址映射；共享只解决可见性，不解决同步”
- 不只写“管道是通信方式”，而要写“管道是内核对象，经文件描述符访问，因此继承 `read/write/close` 的 I/O 语义”

### 3. 概念层和 API 层分开，但边界可调整

一般情况下，理论篇负责解释机制，POSIX/API 篇负责解释接口。

但如果某一主题的理论部分本来就很薄，允许把理论主干和 API 合并在同一篇中，例如 IPC。

判断标准如下：

- 如果理论本身形成独立主线，就拆成“理论篇 + API 篇”
- 如果理论只是 API 的前置概念，不足以撑起单篇，就合并

## 语言要求

### 1. 避免口语化、设问式和说明文腔

避免使用下列表达：

- “先看一下”
- “然后介绍”
- “为什么会这样呢”
- “可以理解成大概就是”
- “放到后面的文章里再说”

更合适的写法是：

- “本篇讨论……”
- “实践部分位于……”
- “核心区别如下”
- “可以压缩成下表”

标题也应避免设问式和口语化，优先使用直接命名。

例如：

- 用“引例 - 为什么需要同步”
- 不用“为什么 mutex 还不够？”

### 2. 避免反复重复同一句判断

若某个关系已经说明，不要再换一种说法重复一遍。

例如：

- 已经说明“共享内存只解决共享，不解决同步”，后文不必再多次用不同措辞重复
- 已经说明“信号更像通知而非数据通道”，后文只需在需要处引用这一边界

### 3. 概念解释必须可落地

涉及机制时，尽量补一句“它在系统里具体对应什么对象”。

例如：

- `pthread_mutex_t` 是匿名同步对象
- `socket` 是通信端点
- `shm_open` 返回的是文件描述符，而不是直接可访问的共享区

## 结构要求

### 1. 章节顺序要服务理解顺序

章节顺序应围绕“先建立问题，再给机制，再给接口，再给边界”组织，而不是按想到什么写什么。

典型顺序：

1. 本篇讨论什么
2. 关键概念或问题
3. 核心机制
4. API 或实现
5. 常见问题 / 边界
6. 小结

### 2. 每篇都应显式说明边界

正文开头应明确：

- 本篇讨论什么
- 本篇不讨论什么
- 后续内容在哪篇继续

例如：

- 同步理论篇不展开 `pthread_*` 细节
- IPC 篇说明本篇聚焦共享内存、管道、FIFO、socket，不扩展消息队列

### 3. 小结只收束，不开新主题

小结只做两件事：

- 压缩本篇主线
- 连接后续文章

不要在小结中突然引入新的机制、接口或例外情况。

## 内容取舍

### 1. 不为“教材完整性”牺牲主线

如果某部分只是教材中的过渡材料、习题背景或推导路径，而不是后续理解的关键支点，应优先删去。

典型可删对象：

- 过长的教材引例
- 只服务于某个算法推导的中间定义
- 不能支撑后续代码的经典软件解法

### 2. 保留能支撑后续代码的概念

应优先保留下列内容：

- 对后续 POSIX 接口有直接支撑的理论
- 对对象关系有澄清作用的层级划分
- 容易误解、但工程上必须明确的边界

例如：

- 共享内存与同步的边界
- 管道关闭端点与 `EOF` 的关系
- 信号是通知，不是数据通道

### 3. 理论段落必须短而硬

大段理论文字应尽量压缩成短段或表格。原则上：

- 先给一句总判断
- 再给一张表或一个对照
- 再补一层最必要的解释

不要把概念篇写成长篇散文。

## 表达形式

### 1. 比较和分类优先用表格

以下内容优先写成表格：

- 类型比较
- 条件 / 性质 / 维度
- 生命周期
- 对象层级
- API 参数与返回值
- 典型适用场景

### 2. 表格要有固定比较维度

表格不应只堆名词，必须体现维度。

常用维度包括：

- 对象
- 所属层级
- 作用
- 生命周期
- 使用范围
- 成功返回
- 失败返回

### 3. 公式和图只用于压缩关键关系

若一个图或公式能显著降低说明成本，就应保留。

例如：

- Amdahl 定律
- 资源分配图的 `G=(V,E)` 形式

但图和公式只应服务主干，不应成为正文主体。

## 理论篇写法

理论篇的目标是把问题空间和机制边界讲清楚。推荐结构如下：

1. 引入问题
2. 主干概念
3. 机制分类
4. 典型问题或副作用
5. 与后续 API 的衔接

理论篇应遵循以下规则：

- 引例只保留一个最典型的
- 经典问题只保留能代表主线的
- 先讲边界，再讲例子
- 不要让教材中的历史性内容压过当前主题

## POSIX / API 篇写法

API 篇不只是列函数名，而要把“接口 - 对象 - 生命周期 - 示例”串起来。推荐结构如下：

1. API 所属对象概览
2. 头文件与返回值约定
3. 使用范围与边界
4. 逐组接口说明
5. 最小示例
6. 生命周期总览或选型总结

### 1. 每组接口都应给出原型

例如：

- `shm_open`
- `mmap`
- `pipe`
- `socket`
- `sigaction`

### 2. 每组接口都应给出参数与返回值表

如果某组接口是本节主角，就应给出：

- 关键参数
- 每个参数的含义
- 成功返回值
- 失败返回值

不要求每个接口都单独占一大节，但不能只给原型而不说明参数和返回值。

### 3. 涉及多个系统对象时，要补层级图或生命周期表

如果主题同时涉及：

- 内核对象
- 文件描述符
- 地址空间映射
- 文件系统名字

就应显式写清各层对象关系。

例如共享内存就应区分：

- `shm object`
- `fd`
- `mmap` 映射

### 4. 示例要能直接跑，且尽量单文件闭环

示例优先满足：

- 单文件可运行
- 不依赖复杂环境
- 数据流清晰

例如：

- 共享内存优先写成 `fork` 版本，而不是拆成两个独立程序
- 管道示例应关闭未使用端
- socket 示例应选最小的 `AF_UNIX` 场景

### 5. 示例不能偷偷依赖未说明前提

如果示例里为了稳定输出用了 `sleep`，必须明确指出：

- 它只是为了演示顺序
- 不是同步手段

## 示例与代码要求

### 1. 示例要服务机制，不要服务“代码量”

示例不是为了展示复杂性，而是为了说明机制。

优先展示：

- 最小数据流
- 最小生命周期
- 最关键的边界条件

### 2. 代码注释只保留必要注释

代码注释应解释机制点，不要解释显而易见的语法。

例如：

- “这里必须先关闭写端，否则读端看不到 EOF”
- “这里 `MAP_SHARED` 才是共享，而不是写时复制”

### 3. 示例中的错误检查可适当收敛

如果某段代码的重点不在错误处理，可以省略大量样板检查，但不能因此模糊主线。

## 术语与概念边界

### 1. 术语要稳定

同一篇中，同一对象应尽量使用同一称呼。

例如：

- “命名管道”与 “FIFO” 可以并列出现，但正文应说明二者对应关系
- “条件变量”与 `condition variable` 的中英文切换要固定

### 2. 要主动指出容易误解的边界

下列边界必须主动说明：

- 共享不等于同步
- 信号是通知，不是数据通道
- 管道是字节流，不保留消息边界
- `shm_unlink` 删除的是名字，不是立即回收对象
- `kill` 不只表示“杀死进程”

### 3. 比较时必须说明“为什么不同”

不要只列出“谁更适合什么”，还要说明差异来源。

例如：

- 为什么 `pthread_*` 更常用于线程同步：因为是匿名对象，要跨进程就必须共享内存
- 为什么命名信号量更适合不相关进程：因为靠名字访问同一内核对象

## 系列衔接与编号

### 1. 每篇都要与前文形成自然连接

开头应指出：

- 这篇承接哪一篇
- 本篇解决上一篇尚未解决的哪个问题

### 2. frontmatter、标题、outline 要统一

需要统一的内容包括：

- 标题命名方式
- 系列编号方式
- `outline` 与正文标题的一致性
- `slug` 与锚点的一致性

不要让正文标题改了、`outline` 还停留在旧结构。

## 常见禁忌

以下情况应尽量避免：

- 把文章写成教材摘录
- 大段口语化说明
- 设问式小标题泛滥
- 只给 API 原型，不给参数和返回值说明
- 示例依赖复杂环境或拆成多个不便运行的程序
- 小结里补新知识点
- 同一概念在一篇里反复换说法

## 最终目标

这组文章的目标不是“覆盖教材”，而是形成一套能够连接操作系统概念、Unix 对象模型和 POSIX 编程接口的笔记体系。

因此，每篇文章都应尽量回答三个问题：

1. 这一主题真正的主干是什么？
2. 它在 Unix / POSIX 中对应哪些对象和接口？
3. 它和前后文章的边界在哪里？
